ITAI 2376 Final Project: Build Your AI Agent
Due: Monday, May 4, 2026, by 11:59 PM Total Points: 100

Overview
In the Midterm, you designed an AI agent on paper. Now you will build it. Your Final is the working version of the blueprint you submitted, implemented in Python and deployed to a GitHub repository.

You can stay with the path you chose in your Midterm, or you can switch. If you switch, explain why in your README.

Choose Your Path
Option A: Single AI Agent (recommended for most students). Build a fully functional agent using LangChain, the OpenAI Agents SDK, LlamaIndex, or another framework of your choice. Your agent must use at least 2 tools, have a reasoning pattern such as ReAct, Chain of Thought, or Plan and Execute, and include memory or retrieval augmented generation.

Option B: Multi Agent System. Build two or more agents with distinct roles that collaborate. You may use CrewAI, AutoGen, LangGraph, or another framework of your choice. Each agent must have its own clearly defined role, tools, and responsibilities.

Both options must show a clear connection to the deep learning concepts we covered this semester.

What to Submit on Canvas

Only one team member submits for the entire group. If you are working solo, you are submitting for yourself.

Optionally, you may also upload a video file of your demo as a Canvas attachment if you want a backup. This is not required, because the demo is already in your repo.

How to Share Your Repository
Your repository must be either public on GitHub, which is the easiest option, or private with Professor McManus added as a collaborator. If you choose the private route, ask in class for the correct GitHub username to invite.

If I cannot open your repository, your project cannot be graded. Please double check the link before submitting.

What Your GitHub Repository Must Contain
This is a full checklist. Because many of you are still learning GitHub, I want to be explicit about every item. If something does not apply to your project, note that in your README instead of leaving it out silently.

1. README.md File
This is the first thing I will read. It must include all of the following items. Project title and a one sentence description of what your agent does. Team member names and roles, or the phrase Solo Project followed by your name. The problem your agent solves and who the target user is. The option you chose, either Single Agent or Multi Agent, with a note if you switched from your Midterm plan. An architecture overview with a short paragraph describing how the agent works, plus the architecture diagram embedded as an image. A list of frameworks and tools used, such as LangChain, CrewAI, your LLM provider, your vector store, and any APIs. Step by step installation instructions including the Python version, how to install dependencies, and how to set up environment variables. How to run the agent, with the exact command or notebook cell someone would use to start it. Example usage with at least 2 or 3 sample inputs and what the agent outputs for each. A section on known limitations describing what your agent cannot do yet and why. A link to the demo video, either linked from inside the repo or to YouTube, Loom, or similar.

2. Your Code
Your code must be organized cleanly. Choose one of these two structures.

If using a Jupyter notebook, place a file called agent.ipynb in the root of the repo.

If using Python scripts, create a folder called src that contains your Python files such as agent.py, tools.py, and memory.py, and place a main.py or run_agent.py file at the root that someone can execute to start the agent.

The code must be clean, readable, and commented. Do not leave dead code, commented out debug lines, or hard coded API keys in the files.

3. requirements.txt File
Create a plain text file called requirements.txt that lists every Python package your agent needs along with the version, so someone else can install everything with the single command pip install -r requirements.txt.

You can generate this file automatically by running pip freeze and redirecting the output into requirements.txt inside your project environment.

4. .env.example File
Create a template file called .env.example that shows what environment variables are needed, with empty values. For example, the file might contain lines such as OPENAI_API_KEY=, ANTHROPIC_API_KEY=, and PINECONE_API_KEY=, each with nothing after the equals sign.

Never commit a real .env file with your actual keys. If you do, your keys become public and can be stolen. Use a .gitignore to exclude it, as described below.

5. .gitignore File
Create a .gitignore file that tells Git what to never upload. At minimum, it should exclude the .env file, the pycache folder, any .pyc files, the .ipynb_checkpoints folder, the venv folder, the .vscode folder, and the .DS_Store file.

6. Architecture Diagram
Include the same diagram from your Midterm blueprint, updated to reflect your final implementation if it changed. Save it as architecture.png or architecture.jpg either in the root of the repo or in a docs folder, and embed it in your README.

7. Demo Inside the Repository
Your demo must be accessible from the repo. Choose one of these three approaches.

The preferred approach is to upload the demo video file directly to the repo in a folder called demo. GitHub supports video files up to 100 MB. If your video is larger, compress it or use one of the other options.

A second approach is to upload the video to YouTube as unlisted, or to Loom or Google Drive, and place the link in your README.

A third approach, if your agent runs in a notebook or has a simple text interface, is to include a recorded screen capture or an animated GIF showing it working.

The demo must show your agent handling at least 3 different real world scenarios and must be 3 to 5 minutes long.

8. Knowledge Base Files
If your agent uses retrieval augmented generation, include the source documents or a sample of them in a folder called data, so I can see what the agent was trained on. If the data is sensitive or too large to include, explain in the README where it comes from and how to obtain it.

9. REFLECTION.md File
Create a file called REFLECTION.md in the root of the repo with the following sections. What worked well in your implementation. What did not work and how you handled it. The biggest technical challenge and how you solved it. If you switched from Single Agent to Multi Agent or vice versa, explain why. What you would build next if you had another semester.

Academic Integrity
You may use large language models and coding assistants to help you build your agent. That is part of the job. However, the design choices, the architecture, the tool selection, and the reflection must be your own thinking. If you copy another team's agent or submit AI generated content as your own analysis in the reflection, that is an integrity violation.

Final Notes
Do not wait until the week of May 3 to learn GitHub. Push early and push often. Every commit is a save point. If something breaks the day before submission, having commit history means you can roll back to a working version.

Grading Rubric
Total Points: 100

Each of the seven criteria below has four performance levels: Excellent, Proficient, Developing, and Beginning. Each level includes the point range and a full description of what earns that level.

Criterion 1: Working Implementation
Worth 30 points

Excellent (27 to 30 points). The agent runs without errors, handles all intended scenarios reliably, produces accurate and useful outputs, recovers gracefully from unexpected inputs, and demonstrates the full reasoning loop of plan, act, observe, and respond.

Proficient (22 to 26 points). The agent runs and completes its main task correctly in most cases. There are minor issues with edge cases or occasional unexpected outputs, but the core functionality is solid.

Developing (15 to 21 points). The agent runs but has noticeable bugs, inconsistent behavior, or limited functionality. It works only for a narrow set of inputs.

Beginning (0 to 14 points). The agent does not run, crashes frequently, or produces outputs that do not match the stated purpose.

Criterion 2: Architecture Quality
Worth 20 points

Excellent (18 to 20 points). Tools, memory, reasoning pattern, and framework choices are all well justified and clearly match the problem. The architecture is coherent, efficient, and reflects mature design thinking. Multi agent systems show clear role separation and communication between agents.

Proficient (14 to 17 points). The architecture is sound and most design choices make sense. One or two components could be better justified or integrated, but the overall structure works.

Developing (9 to 13 points). The architecture has gaps or mismatches between the problem and the chosen components. Some design decisions feel arbitrary or underdeveloped.

Beginning (0 to 8 points). The architecture is unclear, incomplete, or fundamentally mismatched with the agent's purpose. Components are missing or disconnected.

Criterion 3: Deep Learning Connection
Worth 15 points

Excellent (14 to 15 points). The project shows clear and explicit use of deep learning concepts from the course, such as Transformers, embeddings, retrieval augmented generation, or fine tuning. The student explains how each concept is applied and why it matters to the agent.

Proficient (10 to 13 points). Deep learning concepts are used correctly and identified, but the explanation of their role could be deeper or more specific.

Developing (6 to 9 points). Some deep learning concepts appear in the project, but the connection to course content is surface level or unclear.

Beginning (0 to 5 points). There is little or no visible connection to course deep learning content. The agent could have been built without anything learned in ITAI 2376.

Criterion 4: Repository Quality
Worth 10 points

Excellent (9 to 10 points). The repository is professionally organized. The README is complete and clear. The repo includes requirements.txt, .env.example, .gitignore, and a clean file structure. No secrets or keys are committed. The file naming convention is followed exactly.

Proficient (7 to 8 points). The repository is mostly complete with a good README and proper structure. One minor element is missing, such as .env.example or a fully polished file layout. There are no security issues.

Developing (4 to 6 points). The repository is disorganized, is missing key files such as requirements.txt or .gitignore, has a weak README, or does not follow the file naming convention.

Beginning (0 to 3 points). The repository is incomplete, inaccessible, missing a README, contains committed secrets, or ignores the naming convention entirely.

Criterion 5: Demo
Worth 10 points

Excellent (9 to 10 points). The demo is between 3 and 5 minutes long, is clearly narrated, shows the agent handling at least 3 distinct real world scenarios, and highlights both the reasoning process and the final output. Audio and visuals are clear throughout.

Proficient (7 to 8 points). The demo shows the agent working on at least 3 scenarios but could be tighter, better narrated, or clearer in showing the reasoning. The length is within the required range.

Developing (4 to 6 points). The demo is present but too short, too long, missing scenarios, or hard to follow. Narration is unclear or absent.

Beginning (0 to 3 points). The demo is missing, unplayable, or does not show the agent actually running.

Criterion 6: Reflection Document
Worth 10 points

Excellent (9 to 10 points). The reflection is honest, specific, and thoughtful. It clearly addresses what worked, what did not work, the biggest technical challenge, any path change from the Midterm, and what the student would build next. It demonstrates professional maturity.

Proficient (7 to 8 points). The reflection covers all required sections with reasonable detail but could be more specific or introspective in one or two areas.

Developing (4 to 6 points). The reflection is vague, generic, or missing required sections. It reads as an afterthought.

Beginning (0 to 3 points). The reflection is missing or provides almost no substance.

Criterion 7: Presentation of Work
Worth 5 points

Excellent (5 points). The code is clean, readable, well commented, and free of debug clutter. The project feels polished and professional throughout.

Proficient (4 points). The code is readable with minor rough spots. It is mostly polished.

Developing (2 to 3 points). The code is messy, poorly commented, or contains leftover debug prints, hard coded values, or dead code.

Beginning (0 to 1 points). The code is disorganized, unreadable, or shows little effort in presentation.

How Your Final Score Is Calculated
Your total score is the sum of the points earned across all seven criteria. The maximum possible score is 100 points. Each criterion will be scored individually and feedback will be provided for each area so you know exactly where you earned or lost points.
